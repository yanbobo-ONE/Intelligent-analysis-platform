# 智能分析系统实施规划（Phase 详细执行版）

## 0. 规划目标

把“智能分析系统”落成可演示 MVP，覆盖：
1. 前后端基础框架可运行、可测试；
2. 前端 UI 完整可用；
3. 后端真实接口可提供 NL2SQL 能力；
4. 前后端稳定联调并可验收。

---

## 1. 交付边界（MVP）

### 1.1 必做范围
- 会话管理：创建、切换、重命名、删除、历史回放。
- 智能问答：自然语言 -> SQL（只读）-> 查询结果 -> 文本结论。
- 图表展示：柱状图/折线图/饼图 + 表格回退。
- 记忆追问：同会话上下文有效。
- 异常处理：超时、无数据、非法 SQL、服务错误可读提示。

### 1.2 非本期范围
- 多数据库（MySQL/PostgreSQL）
- 权限系统/多租户
- 报表编排与导出中心

---

## 2. 技术实现基线

### 前端
- React + TypeScript + Vite
- Zustand（本地状态）+ React Query（服务端状态）
- ECharts 图表

### 后端
- FastAPI + Uvicorn
- LangChain + Qwen3
- SQLite（业务数据 + 会话数据）

### 关键目录建议
- `backend/app/main.py`
- `backend/app/api/`
- `backend/app/services/`
- `backend/tests/`
- `frontend/src/components/`
- `frontend/src/layout/`
- `frontend/src/services/api.ts`
- `frontend/src/store/`

## Phase 3：后端接口研发（D6-D9）

### 开发任务
- 会话 API：
  - `POST /api/sessions`
  - `GET /api/sessions`
  - `PATCH /api/sessions/{session_id}`
  - `DELETE /api/sessions/{session_id}`
  - `GET /api/sessions/{session_id}/messages`
- 聊天 API：
  - `POST /api/chat`
  - 接入 Qwen3 + LangChain（NL2SQL）
  - SQLite 执行器与结果结构化
  - 输出 `answerText/tableData/chartSpec/trace`
- 记忆策略：recent window + summary。

### P3 统一规范（必须追加到所有 P3 开发项）
- 输入参数必须先做显式校验，禁止把空值、中文占位符、未处理字符串直接传入模型或工具调用。
- 输出参数必须结构化返回，至少保证 `content`、`response_metadata`、`usage_metadata`、`additional_kwargs`、`tool_calls` 的可追踪性。
- 使用 `.env` 中的 `DASHSCOPE_API_KEY` 与模型配置，所有模型调用前先检查 key 是否为空、是否包含非 ASCII 字符、模型名是否可用。
- 流式输出必须保留 chunk 级字段，便于前后端联调时逐块验证。
- 工具调用必须记录实际返回字段结构，不得只记录自然语言内容。
- 所有 P3 接口返回都要统一错误码、错误消息和 trace 字段，避免前端拿到不可解析的数据。
- 涉及 Qwen/ChatOpenAI/LangChain 的封装必须复用 `qwen_client.py`，禁止在业务接口中散写模型初始化逻辑。
- NL2SQL 链路必须先验证 `SQLDatabase`、`SQLDatabaseToolkit`、`create_sql_agent` 的实际输入输出，再允许接入 `/api/chat`。
- 对外返回的 NL2SQL 结果需固定字段：`answer_text`、`table_data`、`chart_spec`、`trace`。
- `trace` 必须至少包含：`model`、`latencyMs`、`toolCalls`、`streaming`。
- 若模型或数据库依赖缺失，必须在启动或测试阶段显式失败，不允许悄悄降级为伪结果。

### 安全稳定任务
- SQL 只读校验（禁 DML/DDL）。
- 表白名单、行数限制、超时机制。
- 统一错误码和可读错误消息。

### 测试任务
- 单测：SQL 拦截、chart 映射、memory 组装。
- 接口测：成功、无数据、非法 SQL、超时、异常兜底。

### 交付物
- 可被前端直接调用的真实 API。
- 接口测试报告。

### 退出标准（DoD）
- 核心接口稳定可用。
- 异常路径可控且可追踪。

## Phase 4：前后端联调（D10-D12）

### 开发任务
- 前后端契约对齐：`answer_text`、`table_data`、`chart_spec`、`trace`、`error`。
- 前端接入后端真实会话 / chat 接口。
- 前端接入流式 chat 协议。
- 图表渲染与 fallback table 联动。

### 测试任务
- 联调冒烟：创建会话 -> 提问 -> 返回文本 + 图表 -> 追问。
- 错误流测试：空输入、非法 SQL、超时、接口异常。
- 性能基线测试：接口延迟、流式首包、SQL 执行耗时。

### 交付物
- 前后端联调可演示版本。
- 联调问题清单与修复结果。

### 退出标准（DoD）
- 主流程闭环可稳定跑通。
- 关键错误路径有明确提示。
